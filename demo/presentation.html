The Problem

🔄

Duplicate Work

Ritik analyzes EOB spikes in Bangalore. Next week, Abhiram does the same analysis for Mumbai — from scratch. Same SQL, same methodology, same 2 hours wasted.

Real example: EOB root cause analysis was done 3 times across the data team in Q1. Each person wrote the same SQL independently.
💨

Knowledge Vanishes

Claude helps you build a query, find a pattern, solve a bug. Session ends — it's gone. The insight lives in one person's head until they forget it too.

No one remembers the runner OT acceptance model's AUC, feature importance, or where the trained model is saved.
⚠️

Bad Data Spreads

"SLA improved" — by how much? For which zone? Based on what query? Vague claims spread as facts. No review, no expiry, no accountability.

"Performance improved significantly" gets forwarded to leadership without any supporting numbers or methodology.
🔒

Teams Are Siloed

Data team discovers rain causes 80% of EOB spikes. Ops team doesn't know. They keep treating it as an attendance problem. Weeks of misaligned effort.

The ops playbook for rain days was only written after the data team's analysis — but it took 3 weeks to connect the two teams.

How CCP Works

One config file. One command. Two layers of AI review. Role-based access.
1
Do your work in Claude Code as usual — analysis, queries, debugging
2
Save with one command /kb EOB Spike Analysis --team data --type analysis
3
Secrets/PII scan — blocks API keys, tokens, Aadhaar numbers, phone numbers. Auto-redacts warnings.
4
AI quality review — Haiku checks locally before commit. Sonnet reviews on PR. Blocks vague claims, asks for evidence.
5
Role-gated save — Admins commit directly. Contributors raise PRs. INDEX.md and MASTER-CONTEXT.md auto-update.

Rain Clusters — Evidence

SELECT cluster_name, AVG(CASE WHEN job_date = '2026-05-03' THEN eob_minutes END) baseline, AVG(CASE WHEN job_date = '2026-05-04' THEN eob_minutes END) spike FROM analytics.job_eob_daily WHERE cluster_name IN ('Bellandur','Sarjapur','Varthur') GROUP BY 1
